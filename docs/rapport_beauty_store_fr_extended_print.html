Introduction

1. Introduction générale

Beauty Store est une application web de commerce electronique specialisee dans la vente de produits de beaute et de soin. Le site permet au client de consulter un catalogue, de voir les details des produits, d'ajouter des articles au panier, de passer une commande et de suivre son etat.

Le projet ne se limite pas a une simple vitrine. Il integre aussi une partie administration permettant au responsable du magasin de gerer le catalogue, les clients, les commandes et les paiements. Cette separation entre interface client et interface administrateur rend le systeme plus complet et plus proche d'une application professionnelle.

Le choix de Bankily B-pay est important, car il adapte le projet au contexte local. Au lieu d'utiliser uniquement des solutions internationales comme Stripe, le projet propose un flux de paiement manuel avec validation par l'administrateur.

Le but principal est de proposer une solution simple pour le client, mais assez complete pour le gestionnaire du magasin.
DécisionConstruire une boutique complète, pas seulement une page vitrine.

Le projet couvre le catalogue, le panier, les commandes, le paiement et l'administration.

ImpactUne base utilisable pour un vrai commerce local.

Le magasin peut recevoir des commandes et suivre les paiements dans un espace dédié.

Contexte

2. Contexte et problématique

Dans beaucoup de petits commerces, la vente en ligne reste difficile a mettre en place. Le commercant a besoin d'une solution simple pour publier ses produits, recevoir des commandes et verifier les paiements. Le client, lui, souhaite une experience rapide et claire.

Le probleme principal est donc de construire un site qui soit facile a utiliser, sans compliquer le parcours d'achat. Le client doit pouvoir parcourir les produits sans etre oblige de creer un compte immediatement. La connexion devient necessaire seulement au moment du checkout, lorsque le site doit enregistrer l'identite du client et son adresse.

Un autre probleme est le paiement. Les solutions automatiques ne sont pas toujours disponibles ou faciles a integrer. Pour cette raison, le projet utilise un systeme Bankily B-pay manuel : le client paie, puis saisit un code que l'administrateur peut verifier.

Besoin clientUn achat rapide sans inscription obligatoire au départ.

Le compte devient nécessaire seulement quand la commande doit être enregistrée.

Besoin adminUn contrôle clair sur les commandes et les paiements.

Le dashboard donne les informations nécessaires pour accepter ou refuser un paiement.

Objectifs techniques

4. Objectifs techniques

Le projet doit etre organise de maniere claire pour faciliter la maintenance. Le backend doit exposer des routes API propres, tandis que le frontend doit consommer ces routes avec un client API centralise.

  • Prevoir une structure qui peut evoluer vers le deploiement.

Ces objectifs techniques evitent d'avoir un projet difficile a modifier. Ils donnent aussi une base plus solide pour le travail en equipe.

Architecture

6. Architecture globale

L'architecture repose sur une communication entre le frontend React et le backend FastAPI. Le frontend envoie des requetes HTTP vers les endpoints du backend. Le backend traite les donnees, communique avec MySQL, puis retourne une reponse JSON.

Le client utilise l'interface React.
React appelle une route API via le client API partage.
FastAPI valide la demande et applique la logique metier.
SQLAlchemy lit ou modifie la base MySQL.
La reponse JSON est retournee au frontend.
L'interface est mise a jour pour l'utilisateur.

Cette architecture separe correctement l'affichage, la logique metier et le stockage.

Flux principalLe navigateur ne parle jamais directement à la base.

Il passe par l'API, ce qui permet de valider et sécuriser les actions.

RésultatUne séparation propre entre interface, logique et données.

Cette séparation simplifie le debug et les futures améliorations.

Produits

12. Gestion des produits

Les produits sont au centre du projet. Chaque produit possede un nom, une description, un prix, un stock, une categorie et une image.

Les images produits sont stockees cote backend dans backend/uploads/products. La base de donnees ne garde que le chemin, comme /uploads/products/hydra-gentle-cleansing-foam.png.

Cette approche facilite l'affichage dans le frontend et evite de stocker les fichiers binaires dans la base. Elle reste simple pour un premier deploiement.

Pour un projet plus grand, un stockage cloud comme Cloudinary ou S3 serait plus robuste.
CatalogueLes produits viennent de la base principale.

MaintenanceChanger une image ne demande pas de modifier React.

L'administrateur peut gérer les produits côté backend.

Panier

14. Logique du panier

Le panier a ete concu pour fonctionner meme sans compte utilisateur. Si le visiteur n'est pas connecte, les produits ajoutes sont conserves dans le navigateur. Cela permet de ne pas bloquer le client au debut du parcours.

Quand l'utilisateur se connecte, le panier local peut etre synchronise avec le backend. Cette logique donne une experience plus fluide, car le client ne perd pas les produits selectionnes avant la connexion.

  • Ajouter un produit au panier.
  • Modifier la quantite.
  • Supprimer un article.
  • Calculer le total.
  • Passer au checkout.
Panier invitéLe visiteur peut préparer son achat sans compte.

Cette décision améliore l'expérience et réduit l'abandon au début.

SynchronisationLe panier peut ensuite être envoyé au backend.

Après connexion, la commande peut être créée avec les articles sélectionnés.

Checkout

15. Checkout et création de commande

Le checkout est l'etape ou le client doit etre connecte. Cette obligation est logique, car il faut enregistrer la commande avec un utilisateur, une adresse et un contact.

Le client saisit son adresse, sa ville, son numero Bankily et le code B-pay. Le backend utilise ces informations pour creer une commande et un paiement en attente de verification.

Apres creation de la commande, le panier est vide et le client peut consulter le suivi dans My Orders.

Le numero Bankily est demande au checkout afin de faciliter la verification du paiement.
ValidationLes informations de livraison sont collectées au bon moment.

Adresse, téléphone et code B-pay deviennent nécessaires seulement à la finalisation.

Après commandeLe client retrouve sa commande dans My Orders.

Il peut suivre le statut sans contacter directement le magasin.

Paiement

16. Paiement Bankily B-pay

Le paiement Bankily B-pay n'est pas automatique dans cette version. Le client paie depuis l'application Bankily, puis entre le code B-pay dans le site.

Le backend valide que le code contient quatre chiffres, puis cree un enregistrement de paiement avec le statut under_review. L'administrateur doit ensuite comparer les informations et accepter ou refuser le paiement.

StatutSignification
under_reviewLe paiement attend la validation admin.
successLe paiement est accepte.
rejectedLe paiement est refuse.
Contrôle manuelL'admin garde la décision finale.

Le système convient à un paiement local sans API automatique.

Règle importanteLe code B-pay doit contenir quatre chiffres.

Cette validation évite les entrées inutilisables avant la revue admin.

Securite

20. Sécurité de l'application

La securite est presente a plusieurs niveaux. Les mots de passe ne sont pas stockes en clair : ils sont haches avec bcrypt. L'authentification utilise JWT, ce qui permet au frontend d'envoyer un token aux routes protegees.

  • Protection des routes admin.
  • Validation des donnees avec Pydantic.
  • Limitation simple sur login/register.
  • Logout avec revocation du token en memoire.
  • Restriction du type et de la taille des images.
Pour une production avec plusieurs serveurs, il est recommande d'utiliser Redis pour les tokens revoques et le rate limit.
AuthentificationLes routes protégées utilisent JWT.

Le frontend envoie le token pour accéder aux informations privées.

ProtectionLes données entrantes sont validées.

Pydantic limite les erreurs de format avant la logique métier.

Deploiement

23. Préparation au déploiement

Pour deployer le projet, il faut preparer le backend, le frontend, la base de donnees et le dossier des images. Le backend doit avoir un fichier d'environnement de production avec la bonne URL MySQL, la cle secrete et les informations Bankily.

Le frontend doit etre construit avec npm run build. Le dossier genere peut ensuite etre servi par un hebergeur frontend ou par un serveur web.

  • Configurer DATABASE_URL.
  • Executer les migrations Alembic.
  • Verifier le dossier uploads/products.
  • Construire le frontend.
  • Configurer CORS selon le domaine final.
BackendLa production demande des variables d'environnement propres.

Les secrets, la base et CORS ne doivent pas rester en configuration locale.

FrontendLe build doit pointer vers l'URL API finale.

La variable VITE_API_URL devient importante au déploiement.

Limites

24. Limites actuelles

Le projet est fonctionnel, mais certaines limites restent presentes. Le paiement Bankily est encore manuel. Cela fonctionne pour un petit magasin, mais demande une verification humaine.

Le stockage des images dans le dossier backend est simple, mais il faudra prevoir des sauvegardes si le projet passe en production. Une solution cloud serait plus securisee a long terme.

  • Paiement non automatique.
  • Dashboard encore perfectible.
  • Besoin de plus de tests end-to-end.
  • Besoin d'une strategie de sauvegarde des images.
  • Besoin d'une configuration production plus complete.
Risque principalLe paiement manuel demande une action humaine.

C'est acceptable au début, mais moins efficace quand le nombre de commandes augmente.

ImagesLes fichiers doivent être sauvegardés.

En production, perdre le dossier uploads signifie perdre les images produits.
